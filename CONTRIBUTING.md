## Getting Started

### Prerequisites

- **Node.js** 20+
- **npm**
- **Docker** (for local PostgreSQL)
- **Git**

### One-Time Setup

```bash
# Fork the repo, then clone your fork
git clone https://github.com/YOUR_USERNAME/grokreview.git
cd grokreview

# Install dependencies
npm install

# Start PostgreSQL
docker compose up -d

# Run database migrations
npx prisma migrate dev

# Copy environment file
cp .env.example .env
# Fill in at least the required keys (see README)
```

## Development Setup

### Environment Variables

At minimum, you'll need:

| Variable | Purpose | Where to get it |
|----------|---------|-----------------|
| `DATABASE_URL` | PostgreSQL connection | `docker compose up -d` provides localhost:5438 |
| `BETTER_AUTH_SECRET` | Session encryption | Run `openssl rand -hex 32` |
| `BETTER_AUTH_URL` | App URL | `http://localhost:3000` for dev |

For AI features, add at least one:

| Variable | Provider | Free tier? |
|----------|----------|-----------|
| `GROQ_API_KEY` | Groq (fastest) | ✅ Yes |
| `MISTRAL_API_KEY` | Mistral | ✅ Yes |
| `HUGGINGFACE_API_KEY` | HuggingFace | ✅ Yes |
| `OPENROUTER_API_KEY` | OpenRouter (200+ models) | ✅ Yes |
| `OLLAMA_BASE_URL` | Ollama (local) | ✅ Free |

### Running the Dev Server

```bash
npm run dev
# Open http://localhost:3000
```

The app auto-reloads on file changes.

### CLI Tool

```bash
cd cli
npm install
npm run build
npm link  # or: npm install -g .
```

## Project Structure

```
├── app/                    # Next.js App Router (pages + API routes)
│   ├── (auth)/             # Auth pages
│   └── (protected)/dashboard/  # Dashboard pages
├── features/               # Feature modules (colocated logic)
│   ├── ai/                 # Multi-provider AI system
│   ├── reviews/            # Review pipeline, history, comparison
│   ├── analytics/          # Analytics, heatmap, leaderboard
│   ├── billing/            # Subscriptions & usage
│   ├── github/             # GitHub integration
│   └── ...                 # Other feature modules
├── components/             # Shared UI components (shadcn/ui)
├── lib/                    # Shared utilities
├── cli/                    # CLI tool (pr-review)
│   └── src/commands/       # Individual commands
└── prisma/                 # Database schema & migrations
```

### Key Patterns

- **Feature modules** in `features/` colocate server logic, components, and types
- **API routes** in `app/api/` — thin wrappers that delegate to feature modules
- **UI components** in `components/ui/` are generated by shadcn/ui
- **Database queries** live in `features/*/server/` files, not inline in components

## Coding Conventions

### TypeScript

- Use **strict TypeScript** — no `any` unless absolutely necessary
- Prefer `type` over `interface` for props and simple shapes
- Use `interface` for public API types that may be extended
- Use `const` assertions (`as const`) for literal types
- Use `@/` path aliases for all imports (e.g., `@/features/reviews/server/...`)

### React & Next.js

- **Server components by default** — only add `"use client"` when you need interactivity
- Use `async` components for data fetching rather than `useEffect` + state
- Colocate page components in `app/` with feature logic in `features/`
- Use `shadcn/ui` components for UI; extend via `cn()` utility
- Prefer Server Actions or API routes over client-side fetch for mutations

### Naming

| Convention | Example |
|-----------|---------|
| Files: `kebab-case.ts` | `get-pr-history.ts` |
| Components: `PascalCase.tsx` | `PRHistoryList.tsx` |
| Functions: `camelCase()` | `getDashboardStats()` |
| Types: `PascalCase` | `DashboardStats` |
| Server functions: prefixed with `get`/`create`/`update`/`delete` | `getUserById()` |

### Database

- Always use Prisma's `@@map` and `@map` for table/column naming (snake_case in DB)
- Migrations are version-controlled in `prisma/migrations/`
- Never edit existing migration files — create new ones via `npx prisma migrate dev`

### AI Providers

When adding a new AI provider:

1. Add provider config to `features/ai/types.ts`
2. Add provider adapter in `features/ai/providers/`
3. Register in `features/ai/registry.ts`
4. Add model costs to `features/usage/server/usage-stats.ts` (if applicable)

### Security rules and framework detection (three copies, kept intentionally separate)

Two bits of logic are independently reimplemented, not shared, across the web app, CLI, and MCP
server:

- Regex-based secret/vulnerability rules: `features/security/rules/{patterns,secrets}.ts` (web
  app), `cli/src/utils/security-rules.ts` (published `grokreview` CLI), `mcp/src/security.ts`
  (published `grokreview-mcp` server).
- Test-framework detection by file extension: `features/test-gen/server/detect-framework.ts`
  (web app), `cli/src/utils/detect-framework.ts` (CLI), inlined in
  `mcp/src/tools/generate-tests.ts` (MCP).

These are **not** meant to be unified into a shared package right now. The CLI and MCP server
are already-published, standalone npm packages with their own build pipelines (and their own
hard-won AI-SDK version pinning and bundling fixes — see `CASE_STUDY.md`); the web app's copies
import `@/` path-aliased types and Prisma-adjacent code that the standalone packages
deliberately can't depend on, and each package's version has already diverged in shape (e.g. the
CLI/MCP security rules combine secrets+patterns into one file with a different return type; the
CLI/MCP framework detection returns a hint string the web app doesn't need). Extracting shared
`@grokreview/security-core` / `@grokreview/testgen-core` packages would mean adding npm workspace
tooling and touching both publish pipelines for the sake of a few hundred lines total that have
drifted only once in this project's history — not worth the risk at this stage. If you add or
change a rule/framework mapping, update all copies in the same PR (a CI check that diffs the rule
arrays across copies would catch drift automatically, if this becomes a recurring problem).

## Testing

```bash
# Run tests once (CI mode)
npm test

# Run tests in watch mode (dev)
npm run test:watch

# Lint
npm run lint

# Type-check
npm run build  # next build includes TypeScript checking
```

- Write tests for utility functions and server logic
- Test files live in `__tests__/` directories next to the code they test
- We use **Vitest** — see `vitest.config.ts` for configuration
- New features should include tests for at least the core logic

## Pull Request Workflow

### Before You Open a PR

1. **Create an issue** first for discussion (unless it's a tiny fix)
2. **Keep PRs focused** — one feature/fix per PR
3. **Write a clear description** — what changed and why
4. **Make sure CI passes** — PRs with failing CI won't be merged

### PR Checklist

```markdown
- [ ] Tests pass (`npm test`)
- [ ] Build passes (`npm run build`)
- [ ] Lint passes (`npm run lint`)
- [ ] No new TypeScript errors
- [ ] Added tests for new functionality
- [ ] Updated documentation (if needed)
```

### After You Open a PR

- CI/CD runs automatically: lint → test → build
- A reviewer will be assigned within a few days
- Address review feedback with additional commits (don't squash until merge)
- Once approved and CI is green, a maintainer will merge

## Branch Strategy

We follow a simplified Git Flow:

```
master  ──── Production-ready code (protected)
develop ──── Integration branch for features
feature/* ── Individual feature branches
fix/* ────── Bug fixes
```

- **`master`** — Protected. Requires CI/CD to pass and a PR review.
- **`develop`** — Integration branch. PRs from feature branches target this.
- **`feature/*`** — Branch off `develop`, name e.g. `feature/add-awesome-thing`
- **`fix/*`** — Branch off `develop`, name e.g. `fix/webhook-timeout`

### Committing

We use conventional commit messages:

```
feat: add multi-model comparison
fix: handle null author in PR history
docs: update API reference
test: add unit tests for usage tracking
chore: upgrade dependencies
```

## Reporting Issues

### Bug Reports

Include:

- A clear, descriptive title
- Steps to reproduce
- Expected vs actual behavior
- Screenshots (if applicable)
- Environment (browser, OS, Node version)

### Feature Requests

Include:

- What problem you're trying to solve
- How you envision the feature working
- Any alternatives you've considered
